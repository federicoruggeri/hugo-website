Intro

The following work is the result of the project activity related to the "Software Systems Engineering" course, taught by professor Antonio Natali at the university of Bologna (Alma Mater Studiorum).

More precisely, as the name tells, the Unity Actor Simulator framework is a very simple package of functionalities that allows to establish a communication with the Unity Engine in order to simulate the QActors' behaviours described in the DSL provided by the professor's framework .

However, it is important to underline that the Unity Actor Simulator's framework preserves its indendepency from the actors' one. Thus, its usage can be exploited for any kind of project that requires to interact with a simulation environment like Unity. The only restriction imposed is that the framework is sensitive to data written in prolog-like syntax, even though even this feature can be overidden and customized (see Project section).
Note
The version of the Unity Editor used is: 5.6.0b4.

Unity Engine

As the name tells, the Unity Actor Simulator project relies upon the Unity Game Engine. Therefore, in order to be able to completely understand all the technical aspects of this work, it is necessary to have a basic knowledge of how the Unity Game Engine works at least. However, whether you are stranger to it or not, the main ideas and features, which describe how this project works, are not tightly coupled to the Unity's specific dialect. On the other hand, since the aim of this introduction page is to give a general view of the Unity Actor Simulator project, it is relevant to spend some words about the Unity Game Engine. Please note that the following content tries to summarize a very small portion of the set of functionalities that this game engine can offer only. More precisely, we will focus on the scripting part. For more information please check the Unity documentation page .
Basic Concepts
We introduce the following main concepts of the Unity Engine.
  • GameObject: GameObjects are the fundamental objects in Unity that represent characters, props and scenery. They do not accomplish much in themselves but they act as containers for Components, which implement the real functionality.

  • Component: base class for everything attached to GameObjects. Components are never created directly via code, but they are simply attached to a GameObject instead. Moreover, Unity allows you to create your own Components using scripts. These allow you to trigger game events, modify Component properties over time and respond to user input in any way you like. The most common example of a Component is the MonoBehaviour class.

  • MonoBehaviour: a MonoBehaviour is the base class from which every Unity script derives. When you use C#, you must explicitly derive from MonoBehaviour. When you use UnityScript (see below), you do not have to explicitly derive from MonoBehaviour.
Scripting Languages
Unity supports two programming languages natively.
  • C#
  • UnityScript: a language designed specifically for use with Unity and modelled after JavaScript.
However, the Unity Engine itself, the core, is written using C\C++ for performance reasons. So how it is possible to code via the above listed scripting languages? The answer is very simple. Unity exploits wrappers in order to access C\C++ libraries via C# or UnityScript. In other terms, wrappers provide cross language interoperability. For more information about this aspect, here it is reported a simple tutorial about creating a C# wrapper for a C\C++ library and importing it in Unity. Lastly, in order to fully understand how scripts are run within the Unity environment, it is important to underline that the game engine uses the Mono development framework.

Vision

The aim of this work is to lay the underpinnings for a virtual simulation environment that, connected to the actors' framework, unburdens the developer from any eventual delay, derived from a tight-coupled interaction with physical devices. Moreover, it reduces the overall experimental costs. Therefore, the question to which we want to give an answer is the following one: "What if the actor is not implemented as a physical device but its behaviour can be injected into a virtual alter-ego?"

Goal

Define a support infrastructure that allows to exchange and process the data (i.e. 3D/2D movement data) coming from an external environment (in this case QActors') and visualize the collected results via a virtual simulation engine with the same expressive power of a properly working physical device. Moreover, labelled a sub-goal, the development process regarding the framework should be organized with respect of factors like code flexibility and maintainability.

Requirements

We want to define a framework that aims to satisfy the following requirements.
Net Communication
Since the QActors communication is tipically done using the TCP protocol, from the communication point of view it is necessary to define the related support in order to setup a channel in which data can be exchanged.
Data Elaboration
Once we have managed to connect the Unity environment with the external world, all the efforts have to be focused on the data elaboration task. In other words, it is necessary to view the collected raw pieces of information from the semantic dimension.
For example: imagine that you want to move forward your unity simulated robot entity and what the net support receives is something like: move -> forward .
How can this piece of information can be mapped into the simulator's engine in order to achieve what we want to see?
Actor's behaviour configuration
Define the support utilities in order to properly link the syntatic dimension with the correct semantic interpretation.
